目的：中間発表後のフィードバックと、その後の設計見直しを反映し、アニレコの解析モード・フレーム保持仕様・ボタン音声仕様を整理する。

## 1. 今回の追加決定

今回追加・更新する仕様は以下の4点。

1. 解析モード名は **高速 / 標準 / 詳細** とする。
2. 「今の何？」を拾うためのローカル保持は、**15秒 × 10fps = 150フレーム** を基本仕様とする。
3. 「アニレコ！」ボタンを押した瞬間に、**「アニレコ！」という音声が鳴る設計** を入れる。
4. **詳細モードでは、動画丸ごと送信ではなく、画像フレームに直前15秒の音声ファイルを補助添付する** 方式を採用する。

重要なのは、**ローカル保持fps** と **AIへ送る枚数** を分けること。

```text
端末内：15秒 × 10fps = 150フレームを保持
AI送信：解析モードに応じて、その中から代表フレームだけを送る
```

常時クラウドへ動画を送るのではなく、端末内に直前15秒のリングバッファを持ち、ボタンを押した瞬間に代表フレームだけを時系列付きでAIへ送信する。動画ファイルをそのままアップロードする方式は、重く、Gemini側の動画理解で映像が低fpsサンプリングされる可能性もあるため、標準設計としては採用しない。

## 2. 15秒 × 10fps を基本にする理由

以前の案では `15秒 × 2fps = 30フレーム` を想定していたが、動物の「今の何？」に答えるには、2fpsでは一瞬の動きが抜ける可能性が高い。

動物行動では、以下の変化が短時間で起きる。

- 魚の急な方向転換
- 鳥の羽ばたき
- 小動物のジャンプ
- 水中への飛び込み
- 威嚇姿勢への移行
- 餌を取る瞬間
- 他個体との接触や距離変化

2fpsは0.5秒に1枚なので、動きの「始まり」と「ピーク」が抜けることがある。  
10fpsは0.1秒に1枚なので、行動の流れをかなり拾いやすい。

前回案は軽量で扱いやすいが、来場者が動きを見てからボタンを押すまでの反応遅れや、行動の前兆を考えると少し余裕がない。  
15秒 × 10fps = 150フレームは保持枚数が増えるものの、これはローカルの一時リングバッファであり、150枚をそのままAIへ送るわけではない。

そのため、現時点の基本仕様は以下とする。

```text
リングバッファ長：15秒
ローカル保持fps：10fps
ローカル保持枚数：150フレーム
標準AI送信枚数：40枚
```

15秒は、前兆・動き始め・ピーク・行動後を拾う安全マージンがあり、展示体験としての取り逃がしを減らせる。

## 3. 解析モード：高速 / 標準 / 詳細

解析モードは、来場者向けUI・開発設定ともに以下の3名称を使う。

| モード名 | 内部名例 | AI送信内容 | 画像サイズ目安 | 位置づけ |
|---|---|---:|---|---|
| 高速 | `FAST` | 18枚 | 640×360 または 768×432 | 低遅延・弱い回線向け |
| 標準 | `STANDARD` | 40枚 | 768×432 または 896×504 | 通常展示の本命 |
| 詳細 | `DETAIL` | 60枚 + 音声補助 | 896×504中心 | 検証・高速な動物・音声手がかりが有効な場合 |

### モードの考え方

3モードで変えるのは、基本的に **AIへ送る代表フレーム数・画像サイズ・圧縮品質・音声補助の有無**。  
ローカル保持は共通で `15秒 × 10fps = 150フレーム` とする。

```text
共通：15秒 × 10fpsで端末内に保持
高速：少ない枚数で素早く返す
標準：速度と精度のバランスを取る
詳細：遅延を許容して動きの判定材料を増やし、必要に応じて音声も補助情報として添付する
```

高速モードでもローカル保持を10fpsにしておくことで、後から代表フレームを選びやすくなる。  
つまり、**10fpsはAI送信量を増やすためではなく、取り逃がしを減らすための保険** として使う。

## 4. 推奨サンプリング

150フレームを均等に間引くだけではなく、ユーザーが押す直前を厚めに見る。

来場者がボタンを押すタイミングは、動物の動きが起きた直後になりやすい。  
そのため、古い時間帯は文脈確認用、直近時間帯は行動判定用として扱う。

### 高速モード：18枚

```text
T-15秒〜T-7秒：5枚
T-7秒〜T-2秒：7枚
T-2秒〜T-0秒：6枚
合計：18枚
```

用途：

- 会場Wi-Fiが弱い
- とにかく応答を早くしたい
- デモで待ち時間を短く見せたい
- 行動が比較的ゆっくりしている

### 標準モード：40枚

```text
T-15秒〜T-7秒：10枚
T-7秒〜T-2秒：16枚
T-2秒〜T-0秒：14枚
合計：40枚
```

用途：

- 通常展示の初期設定
- 速度と判定材料のバランスを取りたい
- 魚・鳥・小動物にもある程度対応したい

### 詳細モード：60枚 + 音声補助

```text
T-15秒〜T-7秒：14枚
T-7秒〜T-2秒：24枚
T-2秒〜T-0秒：22枚
合計：60枚
```

用途：

- 速い動きの検証
- 標準モードで判定が曖昧な場合
- 回線が良い環境
- 開発中の精度確認
- 鳴き声、水しぶき、羽ばたき音などが行動推定に効く場合

詳細モードは標準展示で常用しなくてもよい。  
本番では標準を基本にし、必要時だけ詳細へ切り替える。詳細モードでは、画像60枚に加えて直前15秒の音声ファイルを補助添付できるようにする。

## 5. フレームメタデータ

AIへ送る画像には、必ず時系列情報を付ける。

送信順は **古い → 新しい** に統一する。

各フレームには以下を付ける。

```text
frame_id: frame_01, frame_02, ...
time_offset: T-15.0s, T-10.5s, T-0.0s など
mode: 高速 / 標準 / 詳細
focus: 中央横長フォーカス枠のクロップ画像
```

AIへのプロンプトでは、1枚ずつの画像説明ではなく、**フレーム間の変化から動物の行動を推定する** ように指示する。

## 6. フォーカス枠・クロップ仕様

フォーカス枠は横長16:9を基本にする。

```text
フォーカス比率：16:9
高速モード画像：640×360 または 768×432
標準モード画像：768×432 または 896×504
詳細モード画像：896×504中心
詳細モード音声：直前15秒の音声ファイルを補助添付（ONの場合）
画像形式：JPEG優先
JPEG品質：60〜75
```

フォーカス範囲にカットすること自体は、正しく実装すれば処理を遅らせる主因にはならない。  
むしろ送信ピクセル数が減るため、アップロードとAI処理の面では有利になる。

ただし、以下の実装は避ける。

```text
フル解像度で取得
→ GPUからCPUへ同期読み出し
→ メインスレッドで大量クロップ
→ PNG保存
→ 送信
```

推奨は以下。

```text
10fpsの取得タイミングだけ処理する
RenderTextureまたはプラットフォーム側画像処理でフォーカス範囲を切り出す
AsyncGPUReadback等でメインスレッド停止を避ける
JPEGで軽量化してリングバッファへ入れる
ボタン押下時は、保存済みフレームから代表フレームを選ぶ
```

フォーカス枠は小さすぎないようにする。  
動物の全身、姿勢、周辺の水面・枝・岩・他個体との距離が入る余白を残す。

## 7. 詳細モードの音声補助仕様

動画ファイルをそのままGeminiへアップロードする方式は採用しない。

理由は以下。

- 動画丸ごとは通信量・待ち時間が重くなりやすい
- Geminiの動画理解では映像ストリームが内部的に低fpsサンプリングされるため、速い動きの細部が落ちる可能性がある
- フォーカス枠クロップや、直近側を厚めに見るサンプリング制御が弱くなる
- 会場では音声に周囲の来場者の声・BGM・別展示音が混ざりやすい

そのため、アニレコでは以下を採用する。

```text
動画丸ごと送信：採用しない
基本：フォーカス済み画像フレーム列をGeminiへ送る
詳細モード：画像フレーム列 + 直前15秒の音声ファイルを補助添付する
```

音声は主情報ではなく、あくまで補助情報として扱う。

```text
主情報：横長フォーカス枠から抽出した時系列画像フレーム
補助情報：直前15秒の音声
判断優先度：映像 > 音声
```

音声から期待できる手がかりは以下。

- 鳴き声
- 水しぶきの音
- 羽ばたき音
- 足音、接触音、衝突音
- 餌を食べる音
- 威嚇や反応の声

ただし、実展示では周囲音が混ざる。  
そのため、プロンプトでは人の会話・会場BGM・周囲の雑音を無視し、映像と矛盾する音声は採用しないように指示する。

### 音声バッファ

詳細モードでは、映像リングバッファと同じ15秒幅で音声も一時保持できる設計にする。

```text
音声バッファ長：15秒
用途：詳細モード時のみAIへ補助添付
形式候補：m4a / webm-opus / wav
論理名：analysis_audio_15s
送信単位：直前15秒の音声1ファイル
```

初期実装では、音声解析は必須にしない。  
音声取得や添付に失敗した場合でも、画像フレームだけで解析を続行する。

### デモ時の音源の考え方

物理iPadで固定クリップを再生する展示では、Quest等のマイクで拾う音は周囲音と混ざる。  
そのため、安定性を優先するデモでは、可能なら固定クリップの元動画から該当15秒の音声を切り出す方式も検討する。

ただし、将来の実環境に近い検証では、端末マイクで拾った環境音を使う。  
この場合は、周囲音が混ざることを前提に、音声は補助情報として扱う。

### 「アニレコ！」ボタン音声との分離

ボタン押下時の「アニレコ！」音声は、解析対象の音声とは別物である。

詳細モードでは、ボタン押下時に以下の順序にする。

```text
1. 直前15秒の映像フレームバッファを確定
2. 直前15秒の解析用音声バッファを確定
3. その後で「アニレコ！」の演出音声を再生
4. 画像フレーム + 音声ファイルをGeminiへ送信
```

これにより、演出用の「アニレコ！」音声が解析対象音声に混ざるのを避ける。

## 8. 「アニレコ！」ボタン音声仕様

「アニレコ！」ボタンを押したとき、即座に **「アニレコ！」という音声** を再生する。

この段階では、実際の音源制作までは必須ではない。  
ただし、後から音源を差し込める設計にしておく。

### 目的

- ボタンを押した手応えを出す
- 体験名を印象づける
- 解析開始の合図にする
- AI応答待ちの無音感を減らす

### 再生タイミング

```text
ユーザーが「アニレコ！」ボタンを押す
→ 直前15秒の映像フレームを確定
→ 詳細モードでは直前15秒の解析用音声も確定
→ すぐに「アニレコ！」音声を再生
→ 同時に解析開始UIへ移行
→ フレーム選択・AI送信を開始
```

音声再生はAI処理を待たない。  
ただし詳細モードで解析用音声を使う場合は、先に直前15秒の音声バッファを確定してから演出音声を鳴らす。ボタンを押した瞬間のローカル演出として扱う。

### 仕様

```text
音声名：アニレコ！
用途：ボタン押下時のワンショットSE/ボイス
再生タイミング：onClick / onPressed直後
再生方式：非同期・ローカル再生
音声が無い場合：無音で続行し、解析処理を止めない
連打対策：解析中はボタンを無効化、または短時間デバウンス
```

### 推奨アセット名

OSやビルド環境で扱いやすいよう、ファイル名は英数字・小文字・アンダースコアにする。

```text
表示名：アニレコ！
論理名：anireco_call
推奨ファイル名：anireco_call.wav / anireco_call.mp3 / anireco_call.ogg
```

### Android側の配置案

Androidネイティブ/Androidアプリ側では、将来的に以下へ配置する想定。

```text
app/src/main/res/raw/anireco_call.wav
```

Androidの `res/raw` に入れる場合、ファイル名は小文字英数字とアンダースコアにする。  
`アニレコ！.wav` のような日本語名は避ける。

### Meta / Unity側の配置案

Unity / Meta Quest側では、将来的に以下へ配置する想定。

```text
Assets/Audio/anireco_call.wav
```

実装イメージ：

```text
AudioClip: anireco_call
AudioSource: UI用またはSE用AudioSource
Button押下時: AudioSource.PlayOneShot(anireco_call)
```

音声は空間音響ではなく、UI効果音として聞こえればよい。  
必要ならヘッドロックUIに紐づく2Dサウンドとして扱う。

### Web / anireco-site側の配置案

Webサイト側では、将来的に以下へ配置する想定。

```text
assets/audio/anireco_call.mp3
```

実装イメージ：

```text
const audio = new Audio('assets/audio/anireco_call.mp3');
button.addEventListener('click', () => {
  audio.currentTime = 0;
  audio.play().catch(() => {});
});
```

ブラウザではユーザー操作なしの自動再生が制限されるが、ボタンクリックに紐づく再生なので成立しやすい。

## 10. AIプロンプト方針

AIへは以下の考え方で送る。

```text
以下の画像は、ユーザーが「アニレコ！」ボタンを押す直前15秒間から抽出した時系列フレームです。
frame_01 が最も古く、番号が大きいほど新しい画像です。
各フレームには T-何秒 の情報があります。
中央の横長フォーカス枠内の動物だけを対象にしてください。
背景、来場者、枠外の動物は無視してください。
1枚ずつ説明せず、フレーム間の変化から直前に起きた動物の行動を推定してください。
詳細モードで音声が添付されている場合、音声は補助情報としてのみ使ってください。鳴き声、水音、羽ばたき音、接触音などが行動推定に役立つ場合だけ参照し、人の会話・会場BGM・周囲の雑音は無視してください。映像と音声が矛盾する場合は映像を優先してください。
来場者向けに1〜2文で短く説明してください。
確信が低い場合は断定せず「判定しきれません。もう一度、動物を枠に入れて試してください。」と返してください。
```

内部的にはJSONで扱う。

```json
{
  "behavior": "行動名",
  "explanation": "来場者向けの1〜2文解説",
  "confidence": "high|medium|low",
  "reason": "どのフレーム変化、または補助音声のどの手がかりから判断したか",
  "quiz": {
    "question": "同じ動物に関する別知識の3択クイズ",
    "choices": ["A...", "B...", "C..."],
    "answer": "A",
    "comment": "正解後の短い補足"
  }
}
```

来場者には `explanation` とクイズ部分だけを見せる。  
`confidence` と `reason` はデバッグ用に保持する。

## 11. キャッシュ・フォールバック

ライブAIが遅い、不安定、または低信頼の場合は、事前生成キャッシュに切り替える。

```text
ライブAI成功：AI回答を表示
ライブAIタイムアウト：キャッシュ回答を表示
低信頼：断定せず再試行を促す、またはキャッシュに切替
ネットワークなし：キャッシュ回答のみで体験を継続
```

展示では「絶対に止まらない」ことを優先する。  
音声再生やボタン演出はローカルで完結させ、ネットワーク不調の影響を受けないようにする。

## 12. 実装タスク分解

### 共通

- `AnalysisMode` を定義する
  - `FAST` = 高速
  - `STANDARD` = 標準
  - `DETAIL` = 詳細
- モードごとに送信枚数・画像サイズ・JPEG品質・音声補助の有無を設定する
- ローカルリングバッファは `15秒 × 10fps = 150フレーム` を基本にする
- フレームに `frame_id` と `time_offset` を付ける
- 直近側を厚めにサンプリングする
- ボタン押下時に `anireco_call` 音声を再生できる入口を作る
- 詳細モードでは直前15秒の解析用音声 `analysis_audio_15s` を補助添付できる入口を作る
- 動画ファイル丸ごと送信は標準経路にしない

### Android側

- `res/raw/anireco_call.*` を置ける設計にする
- ボタン押下時に音声再生処理を呼べるようにする
- 音源未配置でもクラッシュしない
- 解析モードをUIまたは開発設定で切り替えられるようにする
- 詳細モード時だけ解析用音声ファイルを添付できる設計にする
- 解析用音声が取得できない場合は画像のみで続行する

### Meta / Unity側

- `Assets/Audio/anireco_call.*` を置ける設計にする
- UI用 `AudioSource` で `PlayOneShot` できるようにする
- ボタン押下時に解析用音声バッファを確定してから、演出音声、解析中UI、フレーム送信処理を開始する
- フォーカス枠と実際のクロップ範囲の一致を早期検証する
- 詳細モード時に画像フレーム列 + 直前15秒音声ファイルをGeminiへ送れる設計にする

### anireco-site側

- `assets/audio/anireco_call.*` を置ける設計にする
- ボタンクリックに紐づけて音声再生する
- ブラウザの再生制限に引っかかってもUIが止まらないようにする
- モード説明を `高速 / 標準 / 詳細` に統一する
- 詳細モードの説明に「音声補助」を含める
